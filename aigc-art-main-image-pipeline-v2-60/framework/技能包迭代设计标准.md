# 技能包迭代设计标准

## 1. 文档目的

本文件用于指导 `aigc-art-main-image-pipeline` 系列技能包的后续迭代。

它不是运行时规则，不参与剧本分析、资产抽取、评级、提示词生成或交付质检。它的用途是：在每次修改技能包之前，帮助维护者判断改动是否符合既有框架设计，是否会破坏层级骨架、规则边界、输入输出契约和全局一致性。

后续每次计划新增、删除、合并或修改文件时，应先用本文件进行设计审查。

## 2. 使用方式

每次迭代前，把本文件和本次修改目标一起提供给执行者，并要求先回答：

| 问题 | 必须回答 |
|---|---|
| 本次修改属于哪一类？ | 框架、模块、层级、规则、模板、守卫、输出契约、元信息、其他 |
| 修改目标是什么？ | 解决哪个具体问题，避免泛泛改动 |
| 影响范围是什么？ | 只影响单文件、单层级、单模块，还是需要全局同步 |
| 是否需要升版本号？ | 任何正式变动都需要升版本号 |
| 是否需要新建文件？ | 如果需要，说明文件类型、存放位置、调用层级 |
| 是否存在职责重合？ | 对比现有文件，说明是否冲突或重复 |
| 是否需要同步更新引用？ | 检查 `workflow.md`、层级文件、模板、守卫、`SKILL.md`、`agents/openai.yaml` |
| 是否需要验证？ | 根据本次改动定制验证项，不套用固定回归清单 |

## 3. 总体设计思想

技能包应保持“骨架清晰、规则纯净、调用明确、输出可接续”。

| 原则 | 含义 |
|---|---|
| 框架只管结构 | `framework` 说明流程、模块、层级、输入输出，不承载底层业务规则正文 |
| 规则只管判断 | `references/rules` 只定义判断标准、字段约束、边界和输出要求 |
| 模板只管生成格式 | `references/templates` 只定义提示词或表格的写法 |
| 守卫只管检查 | `references/guards` 只检查风格、语言、编辑约束和输出质量 |
| 层级只管调用 | `framework/layers` 只说明职责、输入、输出、调用文件，不复制规则正文 |
| 模块只管分组 | `framework/modules` 只说明大模块职责和层级范围，不写底层判断细节 |

不要因为某次局部问题，把临时修复口径提升为全局规则。全局原则必须能长期适用于多数迭代场景。

## 4. 文件类型边界

新增或修改文件前，先判断它属于哪一种文件类型。

| 类型 | 位置 | 作用 | 不应该包含 |
|---|---|---|---|
| 入口文件 | `SKILL.md` | 技能触发描述、简要流程、资源导航 | 大段底层规则、案例堆叠、维护过程记录 |
| 工作流文件 | `framework/workflow.md` | 全流程图、模块说明、输入输出契约 | 具体评级标准、提示词模板、规则正文 |
| 模块文件 | `framework/modules/*.md` | 大模块职责、包含哪些层级 | 底层判断细则、字段全集 |
| 层级文件 | `framework/layers/*.md` | 单层职责、输入、调用文件、输出 | 规则正文、模板正文、跨层级重算逻辑 |
| 规则文件 | `references/rules/*.md` | 判断标准、字段边界、输出要求 | 提示词写法、最终排版、框架流程图 |
| 模板文件 | `references/templates/*.md` | 提示词或输出表结构模板 | 资产评级、状态拆分、审计逻辑 |
| 守卫文件 | `references/guards/*.md` | 质量、风格、语言、编辑约束检查 | 资产抽取、评级、注册、映射生产 |
| 元信息文件 | `agents/openai.yaml` | UI 展示名、简介、默认提示 | 业务规则、流程细节 |

如果一个文件同时想做两类事情，优先拆分，而不是继续扩大文件职责。

## 5. 模块与层级纯净原则

每个层级必须只回答它自己的核心问题。

| 层级 | 核心问题 |
|---|---|
| `script-intake.md` | 原始剧本如何切成稳定项目、集、场和来源定位？ |
| `scene-coverage.md` | 每场客观出现了哪些人物、地点、道具和可见线索？ |
| `asset-extraction.md` | 哪些场次事实可以进入候选资产池？ |
| `state-variant.md` | 哪些资产需要拆成可见状态？ |
| `asset-rating.md` | 每个 base asset 的 `S/A/B/C` 等级是什么？ |
| `asset-registry.md` | 候选资产如何合并、注册，并生成稳定资产 ID、状态 ID 和别名归并表？ |
| `candidate-disposition.md` | 每个人物候选和泛称/群体候选最终去向是什么？ |
| `task-manifest-generation.md` | 已注册且应生产的资产如何生成全局任务草稿？ |
| `episode-usage-map.md` | 每集每场如何引用已有任务、状态或可见排除去向？ |
| `character-anchor.md` | 哪些人物需要脸模锚点，状态图如何引用锚点？ |
| `production-audit.md` | 前序结果是否完整、一致、可进入提示词生成？ |
| `prompt-generation.md` | 如何根据已确认任务生成提示词或编辑指令？ |
| `output-contracts.md` | 如何整理成机器长表和人审短表？ |
| `final-quality.md` | 最终交付是否符合输出契约？ |

修改某一层时，应检查是否把其他层的职责带进来了。常见错误包括：

- 在评级层写提示词规则。
- 在状态层判断任务类型。
- 在锚定层重新评级。
- 在审计层直接修复或重写上游结果。
- 在人审短表逻辑里反向修改机器长表。

## 6. 规则文件新增与修改标准

新增规则文件前，必须先判断是否真的需要新文件。

| 判断问题 | 处理方式 |
|---|---|
| 现有规则能否通过补充小节解决？ | 能解决则优先补充现有规则 |
| 新规则是否回答一个独立核心问题？ | 只有独立问题才新建文件 |
| 新规则是否会被某个层级调用？ | 没有调用层级，默认不应进入正式结构 |
| 新规则是否和现有规则冲突？ | 冲突时先合并或重写边界，不并存 |
| 新规则是否只是某次测试的补丁？ | 先作为测试结论记录，不直接升级为全局规则 |

规则文件必须写清：

| 项目 | 要求 |
|---|---|
| 规则目的 | 说明本文件只解决什么问题 |
| 输入依据 | 明确读取哪些上游结果或材料 |
| 输出结果 | 明确输出什么结构或字段 |
| 判断标准 | 写清正向准入或判断规则 |
| 边界 | 写清不做什么，避免越权 |
| 异常处理 | 说明模糊、缺证据、冲突时如何处理 |

## 7. 输入输出契约原则

每个层级必须有明确输入和输出。

修改流程或层级时，必须检查：

| 检查项 | 判断标准 |
|---|---|
| 是否新增输出？ | 新输出必须被后续某层输入接住，或说明它是终点报告 |
| 是否删除输出？ | 所有引用该输出的下游层级都要同步修改 |
| 是否新增输入？ | 输入来源必须明确，不允许凭空读取未定义结果 |
| 是否改变字段名？ | 所有下游引用都要同步改名 |
| 是否改变枚举值？ | 所有规则、模板、守卫、输出契约都要同步检查 |

注意：不要把某次改动中形成的“下游只能读取上游结果”直接抽象为所有场景的全局禁令。是否允许下游复核、审计或读取原文，应由该层级设计目的决定，并在对应层级文件中写清。

## 8. 字段与枚举治理

字段和枚举值必须集中、稳定、可追踪。

| 类型 | 治理要求 |
|---|---|
| `priority_level` | 只允许 `S/A/B/C`，具体评级标准由人物、场景、道具评级规则分别定义 |
| `task_type` | 只允许 `text_to_image/image_text_edit`，由任务类型字段审核规则集中定义 |
| `review_status` | 只允许 `ready/manual_review_required/excluded`，由任务类型字段审核规则集中定义 |
| `anchor_task_id` | 只在角色锚定和机器长表字段中作为锚点依赖字段 |
| `episode_id/scene_id` | 必须可回溯到剧本解析和场次覆盖 |

新增字段前必须回答：

| 问题 | 要求 |
|---|---|
| 字段来源是什么？ | 来自哪个层级或规则 |
| 字段用途是什么？ | 用于判断、生产、审核、映射还是展示 |
| 谁会读取它？ | 明确下游消费者 |
| 是否进入机器长表？ | 如果进入，必须更新任务类型字段审核规则和输出模板 |
| 是否只是临时诊断？ | 临时诊断字段不要进入正式输出契约 |

## 9. 模板、规则、守卫边界

这三类文件最容易混在一起，必须分清：

| 文件类型 | 回答的问题 | 示例 |
|---|---|---|
| 规则 | 该不该、是什么、如何判断 | 是否评级为 A；是否拆状态；是否需要锚点 |
| 模板 | 具体怎么写 | 人物主图提示词怎么组织；机器长表列顺序 |
| 守卫 | 结果是否合格 | 是否写实；是否改变脸；是否输出缺列 |

如果发现模板里在判断评级，应移回规则。

如果发现规则里在写提示词句式，应移回模板。

如果发现守卫里在改变结果，应改成只报错或标记问题。

## 10. 人审短表与机器长表关系

机器长表和人审短表必须保持分工：

| 输出 | 作用 |
|---|---|
| `main-image-production-table.md` | 中台生产和机器读取的完整 source table |
| `main-image-review-table.md` | 给人审核的派生短表 |

人审短表只用于审核阅读，不得反向影响机器长表。

人审短表不应包含完整全局资产状态任务长表；它只保留项目概览、角色身份锚点短表、分集使用映射等人需要快速检查的内容。

如果修改人审短表，必须确认机器长表未被删减、合并、改名或重算。

## 11. 证据链与不确定性

涉及资产、状态、评级、锚定、排除的判断，都应能说明证据来源。

| 场景 | 原则 |
|---|---|
| 同一人物、场景、道具证据不足 | 默认不合并、不升级，保留不确定说明 |
| 泛称人物或泛化空间反复出现 | 不能只因频次认定为同一个资产 |
| 状态证据不足 | 不静默删除，标记不确定或交给对应审计/审核流程 |
| 排除项可能被误认为遗漏 | 保留可见排除说明 |

证据链原则可以长期适用，但具体如何处理应写在对应规则文件中，不要在本文件里替代业务规则。

## 12. 版本管理原则

只要技能包发生正式变动，就必须更新版本号。

| 变动类型 | 示例版本 |
|---|---|
| 大结构变化、框架重构、目录重组 | `2.0 -> 2.5`、`2.5 -> 3.0` |
| 规则体系明显调整、输出契约变化 | `1.1 -> 1.5`、`2.5 -> 2.6` |
| 小规则修订、局部字段调整 | `1.6 -> 1.61`、`2.5 -> 2.51` |
| 文案修补、错别字、注释澄清 | 也需要更新补丁版本，除非明确只是草稿区修改 |

版本号不是按工作量大小随意决定，而是用于让后续测试、回滚和归档能准确定位变化。

正式版本更新时，应同步检查：

- 技能目录名。
- `SKILL.md` frontmatter `name`。
- `agents/openai.yaml` 展示名和描述。
- 文档中的版本描述。
- 仓库归档版本目录。

## 13. 验证原则

验证项必须根据本次改动定制，不使用固定回归测试清单套所有场景。

制定验证项时，先判断改动类型：

| 改动类型 | 应验证 |
|---|---|
| 框架/层级改动 | 输入输出是否接续；调用文件是否正确；是否有孤立输出或未引用文件 |
| 规则改动 | 是否和现有规则冲突；是否越权；是否影响字段或枚举 |
| 模板改动 | 输出格式是否仍被中台或人审读取；提示词字段是否完整 |
| 守卫改动 | 守卫是否只检查不改写；是否会误伤正常输出 |
| 输出契约改动 | 机器长表和人审短表是否分工不变；字段是否同步 |
| 版本改动 | 目录名、frontmatter、agent metadata、仓库归档是否一致 |

某次测试中发现的问题，可以沉淀为该类改动的验证建议，但不要直接升级成所有迭代都必须执行的全局回归清单。

## 15. 迭代后检查表

每次正式修改后，执行者必须检查：

| 检查问题 | 通过标准 |
|---|---|
| 文件是否放在正确目录？ | 框架、规则、模板、守卫没有混放 |
| 文件职责是否单一？ | 一个文件只回答一个核心问题 |
| 调用关系是否清楚？ | 新文件有明确调用层级，或说明它是维护文档 |
| 输入输出是否接续？ | 没有无人读取的流程输出，除非它是终点报告 |
| 字段和枚举是否集中？ | 没有在多个文件分散定义同一枚举 |
| 旧口径是否残留？ | 没有旧任务类型、旧字段或旧规则路径污染 |
| 版本是否同步？ | 目录名、frontmatter、metadata、文档描述一致 |
| 验证是否完成？ | 按本次改动定制的验证项已执行 |

## 16. 可直接复用的迭代提示词

后续可以把下面这段和具体修改目标一起提供给执行者：

```text
请先阅读《技能包迭代设计标准.md》，再评估本次修改。

不要直接动文件，先回答：
1. 本次修改属于框架、模块、层级、规则、模板、守卫、输出契约、元信息中的哪一类？
2. 是否符合技能包保持框架结构、层级骨架纯净的设计思路？
3. 是否需要新增文件？如果需要，文件应放在哪里，由哪个层级调用？
4. 新增或修改内容是否和现有文件职责冲突或重复？
5. 是否需要同步 workflow、layer、template、guard、SKILL.md、agents/openai.yaml？
6. 是否影响机器长表、人审短表或中台字段？
7. 这次改动需要升到哪个版本号？
8. 本次改动应设计哪些专属验证项？

确认设计无冲突后，再制定修改计划并执行。
```
