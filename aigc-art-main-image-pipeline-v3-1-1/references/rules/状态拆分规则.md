# 状态拆分规则

## 规则目的

本文件用于在 base asset 已评级、已注册后，判断一个 base asset 是否需要拆成可见状态资产。

本文件只定义 S/A 资产的状态边界、状态继承、状态 ID 生成和状态线索去向，不定义资产等级、任务字段、输出表结构或提示词模板。

## 输入依据

必须读取前置层级已经产出的结构化结果：

- `global-asset-registry`
- `asset-rating-result`
- `asset-candidate-list`
- `character-evidence-ledger`
- `scene-evidence-ledger`
- `prop-evidence-ledger`
- `scene-coverage-table`

不得跳过前置结果重新抽取资产、重新评级或重新注册资产。状态必须挂到 `global-asset-registry.base_asset_id`。

## 核心规则

状态资产是同一个 S/A base asset 的可见差异生产目标。

拆状态必须同时满足：

1. 差异能在静态主图中看见。
2. 差异改变的是资产自身的外观、空间条件或物体条件。
3. 差异稳定到足以指导生产，不是瞬时动作、表情或镜头角度。
4. 不拆会导致后续生成混淆不同版本。
5. 有剧本证据，或明确标记证据不足。

## 评级后拆分策略

一个 base asset 只能有一个 `priority_level`，所有状态资产继承 base asset 的 `priority_level`。不得对状态重新评级。

自动状态拆分只面向 `priority_level: S/A` 的资产。B/C 资产完全不进入状态拆分层的状态识别、状态线索去向、人工复核或状态任务生成路径。

| base asset 等级 | 状态处理 |
|---|---|
| `S` | 有明确可见状态差异时自动拆状态。 |
| `A` | 有明确可见状态差异时自动拆状态。 |
| `B` | 状态层忽略，不生成状态资产，不写入 `state-clue-disposition-ledger`，不进入状态人工复核。 |
| `C` | 状态层忽略，不生成状态资产，不写入 `state-clue-disposition-ledger`，不进入状态人工复核。 |

B/C 的服装、伤病、脏污、血迹、空间或物体条件变化不建立任何状态级数据。三类 evidence ledger 若因基础资产证据需要保留原文可见事实，也不得被状态层读取为 B/C 状态线索、汇总为状态数据或写入 `state-variant-result`、`state-clue-disposition-ledger`、状态索引、状态任务、状态人工复核或状态审计问题。状态层不得因为 B/C 状态线索缺少去向而阻断。

不得新增 `state_importance`、`key_state`、`minor_state` 等字段。只要状态需要生成，就已经具备生产意义。

## 主体证据门

状态拆分必须先证明状态属于哪个 S/A base asset。

- 人物状态必须有明确主体证据，例如同一句、相邻动作段或人物列表后的叙述能指向具体人物。
- 场景状态必须指向具体空间。
- 道具状态必须指向具体物件。
- 场次中出现“受伤”“脏污”“血迹”“衣衫凌乱”等关键词时，不得把该状态自动分配给本场所有人物。
- 台词冒号左侧的动作、情绪、语气、声音或瞬时反应不得作为状态主体或状态名，例如 `皇帝点头：`、`棠宁喃喃：`、`皇后拍桌：` 只能为原文证据。
- 如果状态主体不清但可能影响 S/A 资产状态判断，才能记录为不确定线索或审计问题；如果只能指向 B/C、泛称、群体、普通背景资产或无法指向 S/A 资产，则本层直接忽略，不写状态线索去向。

## S/A 状态覆盖策略

S/A 资产的状态拆分采用“状态线索 -> 状态簇 -> 状态资产”的三步法：

1. 只收集能指向 S/A base asset 的可见状态线索。
2. 将相近线索合并为可生产的状态簇，避免逐表情、逐动作、逐轻微脏污拆分。
3. 只有状态簇形成明确可生产外观、空间条件或物体条件时，才进入 `state-variant-result`。

### S 级主角覆盖要求

`priority_level: S` 的人物必须执行完整状态覆盖扫描。不得只依赖人物表头或少量关键词；必须读取该人物在 `character-evidence-ledger` 与 `scene-coverage-table` 中的可见外观线索，覆盖以下状态类型：

- 年龄阶段变化，例如幼年、成年、回忆时期。
- 身份/阶层造型变化，例如装疯、夫人常服、宫宴、流民施粥、囚牢/受审、结尾生活状态。
- 服装轮廓显著变化，例如嫁衣/新妇、中衣、轻纱、外袍、大氅、官服或制服。
- 伤病和身体状态，例如红疹咳血、额头流血、胸口中刀、重伤昏迷、毁容、怀孕、湿发、血迹。
- 伪装或欺骗造型，例如装傻、伪病、诱敌、男装、假身份。
- 剧情转折形成的稳定可见状态，例如复仇血泪、崩溃失魂、狼狈逃脱、醉酒。

以上线索不要求重复出现；只要静态主图生产会混淆不同版本，就应拆为状态资产或明确合并到同一状态簇。

### A 级状态覆盖要求

`priority_level: A` 的资产只拆强可见差异、复用价值高或剧情转折明确的状态簇。A 级不要求像 S 级主角一样全量覆盖，但同一 A 级资产的伤病、毁容、身份造型、空间布置或物件状态发生明显生产差异时，应进入状态判断。

### B/C 排除边界

B/C 资产不做状态覆盖扫描。B/C 状态线索不得进入：

- `state-variant-result`
- `state-clue-disposition-ledger`
- 状态索引表
- 状态任务
- `image_text_edit` 锁脸链路
- 状态覆盖 blocking 审计

B/C 基础资产仍按其 base asset 规则生成或排除；状态层不为其保留额外数据。

## 人物状态边界

人物状态只包含人物本体可见外观变化：

- 年龄外观。
- 脸部或胡须等修饰变化。
- 发型或发色。
- 妆容。
- 服装类别、轮廓、制服、伪装、礼服、工服。
- 伤病、疲惫、脏污、血迹、湿发、怀孕、可见身体状态。
- 阶层/身份呈现导致的可见造型变化。
- 闪回/过去时期的可见外观，如童年、青年、老年、过去工装、过去贫穷或富贵造型。

人物状态不包含：

- 拿着某个道具。
- 站在某个地点。
- 争吵、签字、跪地、举报、复仇、受辱、胜利、失败。
- 与另一个人物的关系。
- 没有稳定外观变化的瞬时情绪。
- 语气、声音或说话方式，例如低声、声音沙哑、冷声、喃喃自语。
- 短暂动作或姿态，例如起身、跪下、摇头、点头、拍桌、回头、抓住、退下，除非这些动作造成稳定可见外观结果。
- 只描述眼神、表情或心理反应的短语，例如眼眸温柔、惊喜、害怕、担忧、不甘、惊恐、大惊、轻呼，除非剧本明确要求该表情形成可生产的稳定人物外观状态。
- 多人组合动作，例如“两人一起”“二夫人三夫人”“二夫人三夫人一起”，不得作为人物状态。

## 场景状态边界

场景状态只包含空间条件的可见变化：

- 昼夜或光照条件显著改变空间身份。
- 停电、应急照明、封闭/封锁状态。
- 活动布置、宴席布置、审堂布置、仪式布置。
- 损毁、打砸、火灾、水淹、污染、事故前后。
- 建设/完工状态。
- 拥挤/空置、满载/清空等改变空间功能的状态。
- 闪回/过去时期的空间条件。

场景状态不包含：

- 对白内容。
- 人物情绪。
- 镜头角度。
- 临时站位。
- 不改变空间的小道具摆放。

## 道具状态边界

道具状态只包含物体条件的可见变化：

- 新旧、完整/破损、干净/脏污、潮湿/烧毁/发霉/沾血。
- 封存/打开，且内部结构或内容重要。
- 标签、盖章、签名、指印、封条、液位、可见内容变化。
- 被改造、隐藏、武器化、污染，或转手后带有可见标记。

道具状态不包含：

- 谁拿着它。
- 它被放在哪里。
- 镜头角度。
- 只有象征意义。
- 人物对它的情绪反应。

## 状态命名

使用短、可见、具体的状态名：

- 人物：`角色名--外观（XX）状态`
- 场景：`场景名--初始（XX）状态`
- 道具：`道具名--初始（XX）状态`

`XX` 必须是可见、具体、对生产有用的描述。

状态名称不得把原文中的人物动作、情绪或语气直接拼成任务名。`二夫人--惊喜`、`三夫人--害怕`、`奶娘--担忧`、`萧厌--低声`、`棠宁--喃喃`、`棠宁--大惊`、`棠宁--轻呼`、`皇后--拍桌`、`皇帝--点头`、`宋父--眼眸温柔` 这类状态默认非法；除非有额外证据证明它们对应稳定可见外观变化，否则应被审计阻断。

## 状态复用

同一 base asset 的同一可见状态可以复用同一个状态任务。

状态复用必须同时满足：

1. `base_asset_id` 相同。
2. `state_name` 指向同一种可见状态。
3. `visual_difference` 没有新增生产差异。
4. 状态证据不冲突。
5. 复用范围不超过该 base asset 的 `priority_level` 连续性范围。

只要外观、空间条件或物体条件发生新的可见生产差异，就必须创建新的状态资产。人物状态是否需要锚定图、是否使用 `image_text_edit`、是否引用 `anchor_task_id`，由 `角色锚定规则.md` 和 `任务类型字段审核规则.md` 处理。本文件只判断状态是否应该拆分和复用。

## 输出结果

输出结果固定命名为：

- `state-variant-result`
- `state-clue-disposition-ledger`

`state-variant-result` 只包含确认需要生成或复用的状态资产。状态 ID 在本层生成，并通过 `base_asset_id` 回指 `global-asset-registry`。

`state-clue-disposition-ledger` 只记录 S/A 状态识别范围内的线索去向：采用为状态、复用既有状态、排除为动作/情绪/语气/场面线索、人工复核或阻断复核。B/C、泛称、群体、普通背景资产或无法指向 S/A 资产的状态线索不进入本台账。

## `state-variant-result` 输出要求

每个状态候选至少携带：

- `base_asset_id`
- `state_asset_id`
- `state_name`
- `asset_type`
- `priority_level`
- `state_basis_type`: `appearance`、`space_condition`、`object_condition`
- `state_trigger_scene_ids`
- `episode_list`
- `scene_list`
- `state_evidence_ids`
- `state_evidence`
- `evidence_strength`: `strong/medium/weak/conflict`
- `visual_difference`
- `split_reason`
- `state_uncertainty_note`，如需要

当 `evidence_strength` 为 `weak` 或 `conflict` 时，必须说明主体不确定、状态是否稳定、是否可能误把场级线索分配给人物。主体不明的状态不得进入可生产状态任务。

## `state-clue-disposition-ledger` 字段

| 字段 | 是否必填 | 规则 |
|---|---:|---|
| `clue_id` | 是 | 稳定状态线索 ID。 |
| `raw_clue` | 是 | 原始状态线索文本或摘要。 |
| `asset_type` | 是 | `character`、`scene`、`prop` 或 `unknown`。 |
| `subject_candidate_id` | 条件必填 | 能指向候选主体时填写。 |
| `base_asset_id` | 条件必填 | 线索已挂到注册资产时填写。 |
| `priority_level` | 条件必填 | 已挂到 base asset 时继承等级。 |
| `source_scene_ids` | 是 | 来源场次 ID。 |
| `source_evidence_ids` | 条件必填 | 能回指 evidence ledger 时填写。 |
| `disposition` | 是 | `used_as_state`、`reused_state`、`excluded_non_state`、`manual_review_required`、`blocking_review`。 |
| `state_asset_id` | 条件必填 | `used_as_state` 或 `reused_state` 时填写。 |
| `exclusion_reason` | 条件必填 | 排除、证据保留或送审时填写。 |
| `source_evidence` | 是 | 支持该去向的原文证据。 |
| `uncertainty_note` | 条件必填 | S/A 主体不明、证据冲突或弱证据时填写。 |

`disposition` 不得当作 `task_type`、`review_status` 或人物 `final_disposition` 使用。

## 边界

- 不定义资产等级。
- 不改变 `priority_level`。
- 不判断 `task_type`。
- 不设置 `anchor_task_id`。
- 不设置 `review_status`。
- 不生成任务。
- 不生成提示词。
- 不生成分集映射。
- 不保留 B/C 状态线索。
- 不让 B/C、泛称、群体或普通背景资产的状态线索干扰 S/A 多状态识别。

只使用以上列举过的字段和职责。
